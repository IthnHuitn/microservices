### Задание 1

#### 1. Ключевое решение: Kubernetes как основа оркестрации

Основной выбор: Kubernetes (K8s).
Альтернативы (и почему они не выбраны):

   - Docker Swarm: Проще в освоении и имеет меньшие накладные расходы, но уступает Kubernetes 
   в возможностях автоматического масштабирования, гибкости сетевых политик и размере экосистемы, 
   что критично для сложных систем .

   - HashiCorp Nomad: Эффективен для ресурсоемких задач и простых сценариев, но имеет более узкую 
   экосистему и меньше возможностей "из коробки" для решения всех поставленных задач, особенно в 
   части управления конфигурацией и безопасностью .

#### 2. Обоснование выполнения требований с помощью Kubernetes

   1. Поддержка контейнеров: Kubernetes является основным инструментом для оркестрации контейнеров 
   (например, Docker, containerd) и управляет их жизненным циклом .

   2. Обнаружение сервисов и маршрутизация: Kubernetes предоставляет встроенный механизм Services, 
   который дает каждому микросервису стабильное DNS-имя (например, backend-service.default.svc.cluster.local). 
   Это позволяет сервисам находить друг друга независимо от того, на каких узлах и с какими IP-адресами они запущены . 
   Для маршрутизации внешнего трафика используется Ingress Controller (например, NGINX, Traefik 
   или AWS Application Load Balancer), который является единой точкой входа и терминации TLS .

   3. Горизонтальное масштабирование: Это достигается с помощью ресурса Deployment, 
   где указывается необходимое количество реплик (экземпляров) каждого микросервиса. 
   Изменяя это число, вы масштабируете сервис вручную .

   4. Автоматическое масштабирование: Реализуется с помощью Horizontal Pod Autoscaler (HPA). 
   HPA автоматически увеличивает или уменьшает количество реплик в Deployment на основе 
   наблюдаемых метрик, таких как загрузка CPU, потребление памяти или пользовательские метрики 
   (например, количество запросов в очереди) .

   5. Разделение внешнего и внутреннего доступа:

      -  Внутри системы: Микросервисы общаются друг с другом через внутренние ClusterIP Services, 
      которые доступны только внутри кластера .

      -  Извне системы: Доступ предоставляется только через Ingress Controller (или LoadBalancer Service), 
      который направляет запросы к нужным внутренним сервисам. Бэкенд-сервисы не имеют внешних IP-адресов. 
      Для сегментации трафика и изоляции между сервисами используются Network Policies — это правила-брандмауэры, 
      определяющие, какие pod'ы могут общаться друг с другом .

   6. Конфигурирование и безопасное хранение секретов:

      -  Для обычной конфигурации (безопасной) используется объект ConfigMap .

      -  Для чувствительных данных (пароли, ключи) используется объект Secret . 
      Для повышения безопасности рекомендуется:

            Хранить секреты не в переменных окружения (которые могут попасть в логи), 
            а монтировать их как файлы в pod'ы через VolumeMounts .

            Включать шифрование данных Secret'ов при их хранении в etcd .

            Использовать внешние решения для управления секретами, такие как HashiCorp Vault
             или AWS Secrets Manager, интегрируя их с Kubernetes через Secrets Store CSI Driver. 
             Это позволит не хранить секреты в коде и централизованно управлять доступом к ним .

### Задание 2

#### 1. Конфигурация кластера: Соответствие схеме

Схема "3 шарда × 3 реплики" означает, что кластер состоит из 3 мастер-нод (по одной на каждый шард) 
и 6 реплик (по 2 на каждый мастер). Общее количество узлов — 9.

Ниже приведено соответствие между абстрактными элементами схемы и реальными компонентами Redis.

---
| Элемент на схеме | Реализация в Redis | Количество | Назначение |
|------------------|--------------------|------------|------------|
| Shard 1, 2, 3 | Master Node (Мастер-нода) | 3 | Каждый шард — это мастер-узел, который владеет своей частью ключевого пространства (хэш-слоты). Принимает запросы на запись. |
| Replication, Replica 1, 2, 3 | Replica Node (Реплика-нода) | 6 (по 2 на мастер) | Реплики асинхронно копируют данные со своего мастера. В случае его отказа, одна из реплик автоматически становится новым мастером (обеспечивая высокую доступность). |
| Replication, Replica 1, 2, 3 | Replica Node (Реплика-нода) | 6 (по 2 на мастер) | Реплики асинхронно копируют данные со своего мастера. В случае его отказа, одна из реплик автоматически становится новым мастером (обеспечивая высокую доступность). |
| VM 1, 2, 3 | Виртуальные машины (или физические хосты) | 3 | Это физические или виртуальные серверы, на которых развернуты контейнеры или процессы Redis. Стратегия размещения нод описана ниже. |
| Client	| Клиентское приложение | 1+ | Любой микросервис, который подключается к кластеру. Клиент должен поддерживать протокол Redis Cluster (например, lettuce для Java, ioredis для Node.js, redis-py-cluster для Python). |

#### 2. Стратегия размещения (Распределение по VM)

2. Стратегия размещения (Распределение по VM)

При развертывании крайне важно соблюдать анти-аффинность, чтобы мастер и его реплики 
не оказались на одной виртуальной машине. Это защищает от потери данных при падении 
одного физического хоста.

Оптимальное распределение 9 нод по 3 VM:

   - VM 1: Мастер Shard 1, Реплика Shard 2 (Replica 3), Реплика Shard 3 (Replica 1)

   - VM 2: Мастер Shard 2, Реплика Shard 1 (Replication), Реплика Shard 3 (Replica 3)

   - VM 3: Мастер Shard 3, Реплика Shard 1 (Replica 1), Реплика Shard 2 (Replication)

Такая конфигурация гарантирует, что если выйдет из строя, например, VM 1, то кластер 
потеряет только Мастер Shard 1. Однако его реплики на VM 2 и VM 3 проведут выборы, 
и одна из них станет новым мастером. Кластер продолжит работу, хоть и в деградированном 
состоянии (без реплик для нового мастера), пока вы не восстановите VM 1.


#### 3. Критические настройки для кластера сессий
```conf
# Включаем режим кластера
cluster-enabled yes

# Файл для хранения состояния кластера (автоматически обновляется)
cluster-config-file nodes-6379.conf

# Таймаут для обнаружения отказа мастера (в мс).
# Рекомендую 5000-10000 мс, чтобы избежать ложных переключений при кратковременных сетевых задержках.
cluster-node-timeout 5000

# Количество реплик, которые должны быть доступны для мастера,
# чтобы он мог принимать запросы (защита от split-brain).
# Рекомендуется установить в 1, чтобы мастер не писал данные в одиночестве.
cluster-require-full-coverage no  # Лучше отключить, чтобы кластер работал при потере части шардов.

# Настройки памяти и политика вытеснения (критично для сессий!)
maxmemory 4gb
maxmemory-policy allkeys-lru  # Вытесняет наименее используемые ключи. Идеально для сессий.

# Настройки безопасности (требование Task 1)
requirepass "strong-master-password"  # Пароль для подключения клиентов
masterauth "strong-master-password"   # Тот же пароль для реплик при подключении к мастеру

# Отключаем опасные команды для продакшена
rename-command CONFIG ""
rename-command FLUSHALL ""
rename-command FLUSHDB ""
```

#### 4. Процесс развертывания и создание кластера

1. Установка Redis: На каждой из 9 нод устанавливается Redis 
(рекомендуется использовать официальный Docker-образ или пакетный менеджер ОС).

2. Настройка: Вносим вышеуказанные параметры в файл redis.conf на каждой ноде. 
Каждая нода должна слушать свой уникальный порт (например, 6379, 6380...) 
и иметь уникальный cluster-config-file.

3. Запуск: Запускаем все 9 экземпляров Redis.

4. Создание кластера: Подключаемся к любой ноде с помощью утилиты redis-cli 
и выполняем команду, которая объединит все узлы в кластер и автоматически распределит 
16384 хэш-слота между тремя мастерами.
```bash
redis-cli --cluster create \
  VM1_IP:6379 VM2_IP:6379 VM3_IP:6379 \ # Это мастера (Shard 1, 2, 3)
  VM1_IP:6380 VM2_IP:6380 VM3_IP:6380 \ # Это реплики для Shard 1
  VM1_IP:6381 VM2_IP:6381 VM3_IP:6381 \ # Это реплики для Shard 2
  --cluster-replicas 2 \
  -a "strong-master-password"
```
- Пояснение: Первые три ноды в списке (VM1_IP:6379, VM2_IP:6379, VM3_IP:6379) автоматически станут мастерами. 
Параметр --cluster-replicas 2 указывает, что для каждого мастера нужно назначить по 2 реплики из оставшегося списка.

#### 5. Интеграция с микросервисами и мониторинг

   - Подключение клиентов: Все микросервисы должны подключаться к кластеру, передавая список из всех 9 хостов (или хотя бы адреса всех мастеров). Клиентская библиотека сама определит текущую топологию и будет направлять запросы к нужному мастеру.

   - Мониторинг: Рекомендую настроить сбор метрик с каждой ноды (используя redis_exporter для Prometheus) и дашборд в Grafana для отслеживания:

      -  Использования памяти (чтобы вовремя увеличить maxmemory).

      -  Количества ключей и их времени жизни (TTL).

      -  Состояния репликации и наличия отказавших узлов.

   - Живучесть (Health Checks): Настроить в оркестраторе Kubernetes readinessProbe и livenessProbe. Для Redis Cluster рекомендуется использовать команду PING или CLUSTER NODES, а не проверять только TCP-порт.

Это решение полностью соответствует предоставленной архитектуре, обеспечивает высокую доступность (отказ одного узла и даже целой VM не остановит кластер), горизонтальное масштабирование (добавление новых шардов) и безопасность.
